🎯 2. Três opções para o estado-alvo — qual adotar?

A

Tabia_IS com MESMA interface

RECOMENDADO

Como funciona

Novo Tabia_IS expõe os mesmos nomes de Server Actions, Structures e campos. Por dentro, traduz para a API Tabia.

Impacto nas 5 apps

  • ZERO mudanças de código
  • Trocar só a dependência (Healthmap_IS → Tabia_IS)
  • Rollback fácil
  • Tabia_IS mais complexo (precisa traduzir tudo)
  • Não aproveita campos novos da Tabia (tabiaScore, pathways)

Esforço

Alto no IS, baixo nas apps. Trabalho concentrado em 1 lib.

B

Tabia_IS com nova interface (PT→EN)

Como funciona

Novo Tabia_IS expõe nomes Tabia (GetPatients, PatientDTO). As 5 apps são refatoradas.

Impacto nas 5 apps

  • MUDAR em TODAS as pontas: Screens, Actions, Aggregates, Local Variables, UI bindings
  • Toda referência a .Nome, .CPF, .NumeroCarteirinha vira .FullName, .Extensions.Cpf, .MedicalRecordId
  • Aproveita 100% dos campos novos da Tabia
  • Código final mais limpo
  • Risco alto de regressão

Esforço

Alto no IS + alto nas 5 apps. Multiplica pelo número de telas/actions.

C

Facade + migração gradual

Como funciona

Mantém Healthmap_IS como facade que chama o novo Tabia_IS. Com tempo, cada app migra para usar Tabia_IS direto.

Impacto nas 5 apps

  • Transição gradual, app por app
  • Permite convivência Healthmap + Tabia durante meses
  • 2 libs vivas em paralelo (complexidade operacional)
  • Facade tem mesmo custo da Opção A
  • Útil se as apps forem migrar de verdade para a interface nova no futuro

Esforço

Médio contínuo. Dispersão de esforço por várias sprints.

💡 Recomendação: Opção A para a primeira fase (destravar a migração sem mexer nas 5 apps), e Opção C em médio prazo se fizer sentido modernizar o código das apps com a nomenclatura Tabia.

🔎 6. Como descobrir exatamente onde impacta (ferramentas OutSystems)

Para cada uma das 5 apps front, antes de decidir:

Ferramenta ODCComo usarO que descobrir
Manage Dependencies Abrir a app no Service Studio → botão Manage Dependencies → filtrar por Healthmap_IS Lista de TODAS as Server Actions, Structures e Entities consumidas. Exporte esse relatório.
Find Usages (Ctrl+Shift+U) Em cada Structure importada (ex: BeneficiarioDTO), botão direito → Find Usages Onde cada DTO é usado: variables, aggregates, actions, expressions
True Change Após trocar a dependência, rodar True Change antes de publicar Lista todas as quebras de contrato. Em Opção A: deve ser zero. Em Opção B: centenas.
ODC Portal → Architecture Portal da app → aba Architecture → grafo de dependências Visão macro de quais apps consomem o IS
Refactoring Report Service Studio 11+ tem relatório de impacto Sugere renomear/substituir em lote

Checklist de inventário (rodar em cada uma das 5 apps)

  • Listar todas as Server Actions de Healthmap_IS consumidas (Manage Dependencies)
  • Para cada Server Action, listar em quantas telas/actions ela aparece (Find Usages)
  • Listar todas as Structures importadas de Healthmap_IS
  • Para cada Structure, listar quantas variables/aggregates/expressions a referenciam
  • Identificar campos específicos mais usados (.Nome, .CPF, .NumeroCarteirinha, etc)
  • Contar total de telas impactadas por app
  • Exportar relatório consolidado para comparar Opção A vs B
💡 Dica prática: Peça para cada squad da app rodar esse inventário em 1 dia. Com os números na mesa, a escolha entre A e B fica óbvia.

⚠️ 7. Cuidados mesmo escolhendo a Opção A (mesma interface)

Tabia_IS com mesma interface não significa que tudo vai funcionar igual automaticamente. Atenção para:

#PontoRisco / Ação
1 Campos sem equivalente (idExterno, respostaTexto, respostaValor) Podem vir nulos ou vazios em telas que hoje esperam valor. Validar onde são usados antes.
2 CPF em extensions na Tabia Adapter precisa buscar de person.extensions.cpf. Se extensão não estiver setada, CPF volta vazio.
3 Enums mapeados (ATIVO → OPEN, PENDENTE → closed/accomplished) Se app faz if status = "ATIVO", precisa continuar funcionando. Adapter deve retornar sempre "ATIVO"/"INATIVO" como string.
4 Paginação base 1 vs 0 Healthmap usa base 1, Tabia base 0. Adapter deve ajustar. Validar com paginações grandes.
5 IDs diferentes IDs Healthmap ≠ Tabia. Adapter deve retornar o ID Healthmap (via tabela de-para) — ou validar se as apps armazenam esses IDs em entities locais.
6 Latência ~460ms por chamada. Se app faz 10 chamadas sequenciais, soma 4.6s. Avaliar timeouts e loadings.
7 Termos de Aceite (4 endpoints ausentes na Tabia) Bloqueia Viver Bem. Solução: manter Healthmap_IS temporariamente SÓ para esses 4 endpoints.
8 MedicaoRisco (conceito diferente) Se app mostra statusRisco = "ALTO", o adapter precisa derivar de tabiaScore.severity. Testar.

✅ 9. Resumo decisório (para levar à reunião)

PerguntaResposta
As 5 apps precisam ser alteradas?Opção A: NÃO (só trocar dependência). Opção B: SIM, em todas as pontas.
Vou quebrar nomenclatura de objetos?Só na Opção B. Na Opção A, BeneficiarioDTO.Nome, .CPF continuam iguais.
Preciso refatorar UI?Só na Opção B. Na Opção A, bindings continuam intactos.
E os campos novos que a Tabia oferece?Opção A perde eles. Se quiser aproveitar, aplicar Opção C (migração gradual posterior).
Qual é mais seguro?Opção A — mudança isolada, rollback fácil.
Qual é mais rápido para entrar em produção?Opção A — sem retrabalho nas 5 apps.
E no futuro, se quiser modernizar?Aplicar Opção C por cima da A — migrar apps uma por uma para a interface Tabia.
Recomendação final: Ir de Opção A.
Trabalho concentrado no novo Tabia_IS, apps intactas, rollback trivial. Se no futuro a equipe quiser usar os campos novos da Tabia, migra gradualmente (Opção C).
