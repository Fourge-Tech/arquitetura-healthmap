Integrações CBV com Healthmap

Referência para Migração ao Sistema Tabia
📋 Versão 1.0 📅 28/03/2026 🏥 Unimed POA 🔄 Em elaboração
1

Objetivo

Este documento mapeia todas as integrações existentes entre o sistema CBV (Unimed POA) e a plataforma Healthmap, fornecendo uma base estruturada para a futura migração dessas integrações para o sistema Tabia.

O documento cobre:

  • Inventário completo de APIs Healthmap consumidas pelo Modelo Assistencial
  • Mapeamento por subsistema/módulo
  • Parâmetros, contratos e fluxos de cada integração
  • Endpoints externos identificados (fora do Swagger principal)
  • Considerações e riscos para a migração ao Tabia

7

Padrões Arquiteturais Identificados

#PadrãoDescriçãoImpacto na Migração
7.1 Autenticação OAuth2 Todos os subsistemas compartilham o mesmo endpoint /oauth/token. Header KeyDS distingue os ambientes PRD/HMG. O Tabia precisará oferecer mecanismo equivalente ou será necessário um adapter/gateway de autenticação.
7.2 Versionamento de API Padrão /api/{domínio}/v1/{recurso} em todos os endpoints do Swagger principal. Verificar se o Tabia segue o mesmo padrão ou requer ajuste nas chamadas.
7.3 Paginação Respostas de listagem retornam objeto Page<T> com campos pageSize, offset e pageNumber. Se o Tabia usar paginação por cursor ou modelo diferente, será necessário adaptar a camada de consumo.
7.4 Endpoints Transversais Beneficiário, Risco e Autenticação são utilizados por todos os 6 subsistemas. Alto risco: mudança impacta toda a plataforma simultaneamente. Exige prioridade no planejamento.
7.5 Dois Backends em Produção healthmapv3-api-unimedpa-prd… (principal, Swagger disponível) e unimedpa.healthmap.com.br (complementar, sem Swagger). A migração deve cobrir ambos os backends. O complementar exige levantamento adicional.
8

Considerações e Riscos para Migração ao Tabia

8.1 Riscos Identificados
Risco 1 — Endpoints sem Swagger
Risco
4 endpoints em unimedpa.healthmap.com.br sem contrato formal. Dificulta mapear equivalentes no Tabia.
Mitigação
Solicitar documentação ao time Healthmap antes do início da migração.
Risco 2 — Alta Transversalidade
Risco
Beneficiário e Autenticação usados por todos os 6 subsistemas. Incompatibilidade no Tabia afeta toda a plataforma.
Mitigação
Priorizar validação desses contratos no início do projeto de migração.
Risco 3 — Ambientes PRD vs HMG
Risco
URL de homologação usa subdomínio diferente. O Tabia pode ter estratégia diferente de ambientes.
Mitigação
Mapear estratégia de ambientes no Tabia e ajustar variáveis de configuração.
Risco 4 — Modelos de Dados Distintos
Risco
DTOs Healthmap (Usuario, AcaoPlanoAcaoDTO, etc.) podem ter estrutura diferente no Tabia. Necessidade de transformação de dados.
Mitigação
Mapear campos equivalentes (field mapping) antes da implementação.
8.2 Recomendações para o Projeto de Migração
  1. Levantar contratos completos dos 4 endpoints externos (seção 6) junto ao time Healthmap.
  2. Iniciar validação dos contratos do Tabia pelos endpoints transversais: Autenticação e Beneficiário.
  3. Criar camada de abstração (adapter / anti-corruption layer) para isolar as chamadas ao Healthmap, facilitando a substituição pelo Tabia sem impactar regras de negócio.
  4. Definir estratégia de migração: big bang (troca simultânea) vs. strangler fig (migração gradual por subsistema).
  5. Garantir ambiente de homologação do Tabia disponível para testes antes da virada em produção.
  6. Documentar o field mapping entre os modelos de dados Healthmap e Tabia para cada domínio.

Integração Viver Bem → CRM Dynamics 365

Fluxo de automação de marketing para gestão de beneficiários na Linha de Cuidado. Dados originados no Healthmap são processados pelo Viver Bem, que aciona o CRM Dynamics 365 (Unimed POA) via API REST (Dataverse Web API v9.2). O CRM, por sua vez, dispara notificações via BLIP pelo canal WhatsApp ao beneficiário.

11.1 Autenticação — Azure AD (OAuth 2.0 Client Credentials)

A autenticação com o CRM Dynamics 365 usa o fluxo client_credentials do Azure Active Directory, diferente do OAuth2 padrão Healthmap. O token obtido é enviado como Bearer em todas as chamadas subsequentes à Dataverse Web API.

OperaçãoMétodoEndpointJSON
Obter Token Azure AD POST https://login.windows.net/{tenantid}/oauth2/token
ℹ️ Diferença em relação ao Healthmap O Healthmap autentica via POST /oauth/token na própria plataforma com KeyDS. O CRM usa o Azure AD (login.windows.net/{tenantid}/oauth2/token) com grant_type=client_credentials, client_id, client_secret e o parâmetro resource apontando para a URL base do CRM. O script de teste Postman extrai automaticamente o access_token e salva na variável token da collection.
11.2 Fluxo de Integração — Passo a Passo

O Viver Bem executa os seguintes passos de forma sequencial ao processar um beneficiário para a Linha de Cuidado:

  1. Obter Token — Autentica no Azure AD (POST /oauth2/token) e obtém o Bearer Token do CRM.
  2. Verificar Contato por CPF — Consulta a entidade contacts filtrando pelo campo customizado uni_cpf.
  3. Criar / Atualizar Contato — Se o contato não existir, cria via POST /api/data/v9.2/contacts. Se existir, atualiza telefone e e-mail via PATCH com os dados vindos do Healthmap.
  4. Verificar Linha de Cuidado por CPF — Consulta uni_linhadecuidados expandindo o contato vinculado e filtrando por uni_cpf.
  5. Criar / Atualizar Linha de Cuidado — Se não existir, cria o registro com os flags de monitoramento. Se existir, atualiza os flags via PATCH.
  6. CRM → BLIP → WhatsApp — Automação interna do Dynamics 365 detecta a criação/atualização da Linha de Cuidado e aciona o BLIP para envio da mensagem WhatsApp ao beneficiário.
ℹ️ Rastreamento de mensagens Após o disparo via BLIP, o CRM cria um registro na entidade uni_whatsapps com o status, template utilizado, fila e eventual falha — permitindo auditoria do envio.
11.3 Contato — Beneficiário no CRM contacts

Entidade padrão contacts do Dynamics 365 com campo customizado uni_cpf para identificação do beneficiário Unimed. Os campos relevantes para o fluxo são fullname, telephone1 e emailaddress1.

OperaçãoMétodoEndpointJSON
Obter Por CPF GET /api/data/v9.2/contacts?$filter=uni_cpf eq '{cpf}'
Atualizar Contato PATCH /api/data/v9.2/contacts({contactid})
ℹ️ Criação de Contato A criação de novo contato usa POST /api/data/v9.2/contacts (operação nativa do Dynamics 365, não mapeada explicitamente na collection). O campo uni_cpf deve ser preenchido na criação para viabilizar consultas subsequentes por CPF.
11.4 Linha de Cuidado uni_linhadecuidados

Entidade customizada do CRM que representa o vínculo do beneficiário ao programa de cuidado. Os flags de monitoramento alimentam as automações do Dynamics 365 que acionam o BLIP para disparo de mensagens WhatsApp.

OperaçãoMétodoEndpointJSON

Objetivo e Escopo

Este documento tem como objetivo registrar e transferir o conhecimento técnico do sistema Viver Bem, módulo central do Modelo Assistencial da Unimed Porto Alegre, desenvolvido na plataforma OutSystems ODC.

Serve como guia de repasse para o profissional que assumirá a manutenção e evolução do sistema, cobrindo:

  • Fluxos de autenticação e acesso (SSO UniiD)
  • Jornada completa de onboarding do beneficiário
  • Módulos funcionais: Minha Saúde e Plano de Cuidado
  • Integrações com Healthmap (APIs REST) e Dynamics 365 CRM
  • Contratos de API — endpoints, payloads e regras de negócio associadas
📌 Contexto de uso O Viver Bem é acessado exclusivamente pelo beneficiário através do App Unimed, por meio de Menu ou Banner na Home. Não há acesso web direto — todo o fluxo é orquestrado pelo OutSystems ODC como camada intermediária entre o usuário final e as APIs de backend.

Módulo 1 — Autenticação e Acesso (SSO UniiD)

Visão Geral

O acesso ao Viver Bem é realizado de forma integrada através do App Unimed. A autenticação utiliza o UniiD (Single Sign-On), ferramenta de identidade desenvolvida pela TI da Unimed Porto Alegre.

ℹ️ Responsabilidade do token A geração e troca de tokens de segurança ocorrem exclusivamente entre o App Unimed e o UniiD, sem envolvimento direto do Modelo Assistencial nesta camada de criação. O OutSystems ODC recebe o token já validado.
Fluxo de Acesso
#EtapaAtorDescrição
1EntradaBeneficiárioAcessa via Menu ou Banner Home no App Unimed
2Validação de tokenOutSystems ↔ UniiDO sistema verifica o token de sessão junto ao UniiD
3SucessoOutSystemsToken válido → carrega o Viver Bem diretamente
4Fallback (erro)UniiDToken inválido ou falha → redireciona para tela de login manual (usuário/senha) do UniiD
⚠️ Ponto de atenção para repasse O fluxo de fallback (login manual) é gerenciado inteiramente pelo UniiD. Em caso de problemas de autenticação, o time responsável é a TI da Unimed POA, não o time do Modelo Assistencial.

Módulo 2 — Onboarding

O Onboarding é a etapa obrigatória de recepção do usuário, executada sequencialmente em 3 passos na primeira vez que o beneficiário acessa o Viver Bem.

4.1 Termo de Aceite e Governança Healthmap

Garante a conformidade legal e o registro assistencial centralizado. As chamadas são executadas em sequência:

#AçãoMétodoEndpointJSON
1 Identificar empresa do beneficiário GET /api/beneficiario/v1/beneficiario
2 Buscar termo vigente da empresa (idEmpresa) GET /api/termo/v1/termo/buscaTermoPublicadoPorEmpresa
3 Registrar aceite do beneficiário GET /api/termo/v1/termo/aceiteTermo
ℹ️ Ordem obrigatória O idEmpresa retornado pela API de beneficiário é parâmetro de entrada da busca do termo. As chamadas devem ser feitas nessa sequência.
4.2 Atualização Cadastral Inicial CRM + OutSystems

Fase de conferência de dados e personalização do perfil do beneficiário.

CampoSistemaMétodoAçãoJSON
E-mail e Telefone Dynamics 365 CRM GET Busca e exibe os dados atuais para o beneficiário conferir
E-mail e Telefone (alteração) Dynamics 365 CRM PATCH Se o beneficiário alterar dados, atualiza o CRM em tempo real
Apelido OutSystems (local) — Salvo na base local OutSystems — usado em saudações ("Como quero ser chamado") —
Código de Indicação OutSystems (Backoffice) — Campo opcional — valida consultores externos via módulo Backoffice —
4.3 Quicktour

Demonstração visual rápida das funcionalidades da plataforma, exibida após o cadastral e antes do redirecionamento para a Home. Não consome APIs externas — conteúdo estático parametrizado no OutSystems.

📌 Comportamento O Quicktour é exibido apenas uma vez (primeiro acesso). A flag de controle é salva na base local OutSystems associada ao idBeneficiario.

Roadmap — Comunidades

Funcionalidade planejada para 2025 que transforma o Viver Bem em um ecossistema social dentro do app, com foco em engajamento e escala do autocuidado.

📌 Status atual Telas conceituais em validação de MVP. Não há implementação em produção nesta data. Este registro é para contextualização do repasse.
Visão Estratégica

O modelo de Comunidades busca transformar o cuidado com a saúde em hábito coletivo, criando um ciclo de engajamento crescente:

Benefício EsperadoDescrição
Melhora nos indicadores do Mapa de SaúdeEngajamento social estimula consistência nos questionários e hábitos
Regularidade no cuidadoGrupos e feed incentivam frequência nas atividades do plano
Melhora na percepção de bem-estarSenso de progresso coletivo amplifica resultados individuais
Aumento da autonomiaUsuários mais informados tomam decisões de saúde com mais independência
Sensação de pertencimentoVínculo com a comunidade aumenta retenção na plataforma
5 Pilares da Comunidade Digital
👥 Grupos
Grupos temáticos com agenda, dicas, plano e progresso — ex: grupo de Pilates com horários de aulas presenciais
📰 Feed
Feed social com conteúdos gerados pelos membros da comunidade, estimulando troca de experiências
📈 Progresso Individual
Visualização do nível e frequência pessoal dentro do grupo — ex: "Nível 1 — Iniciante no Pilates — 62% de frequência"
🏆 Troféus
Sistema de conquistas vinculado ao Meu Plano — gamificação do cumprimento de ações de saúde com compartilhamento social
📲 Redes Sociais
Compartilhamento de conquistas (troféus) nas redes sociais do beneficiário com identidade visual Unimed / Viver Bem
⚠️ Ponto de atenção para repasse O módulo de Comunidades ainda não possui APIs definidas. Quando a feature entrar em desenvolvimento, será necessário mapear os endpoints de: criação/listagem de grupos, feed de posts, sistema de pontos/troféus e integração com Healthmap para validação de conquistas.
