🚀 Kickoff da Migração Healthmap → Tabia

Checklist visual do que alinhar antes de codar — use com a Tabia e com o Dev Lead

🏗️ Contexto: Projeto OutSystems ODC (Open Digital Cloud) Full

A migração será 100% em OutSystems ODC. Isso muda a estratégia de arquitetura: não haverá microservico externo — o adapter vira uma Library ODC, consumindo a API REST da Tabia direto do OpenAPI importado no Service Studio.

Adapter (ACL) ODC Library reutilizavel chamada Tabia_ACL_Lib — expoe Server Actions com assinatura identica ao atual ModeloAssistencial_Lib.
Consume REST API Importar openapi-careos-simple.json no Service Studio gera automaticamente Structures + Client Actions para todos os endpoints Tabia.
Autenticação API Key gerenciada via ODC Secret (Settings > Secrets) — acessada via GetEnvironmentSecret(). Não precisa persistir em DB.
Mappers Server Actions dedicadas (ex: Map_PatientDTO_to_BeneficiarioDTO) — puras, sem side effects, facil testar.
De-para de IDs Entity CBV_TabiaIdMapping no modulo da Library com indexes por tipo + valor.
Estratégia Strangler Fig via Site Property (feature flag) por endpoint: UseTabiaFor_Beneficiario = true/false.

📋 Perguntas e validacoes para a Tabia Health 8 itens
1. Suporte a busca por CPF em /apiv1/patient e /apiv1/user

O Healthmap usa CPF como chave primaria. A Tabia busca por email/cellphone. Precisamos saber:

• Posso adicionar cpf como query param nativo?

• Se não, devo usar person.extensions.cpf? Como indexar para performance?

• Onde o CPF fica armazenado em PatientDTO?

Blocker
2. Timeline de entrega dos endpoints de Termos de Aceite

4 endpoints ausentes: aceiteTermo, buscaTermoPublicadoPorEmpresa, buscarAceiteTermo, revogaTermo. Bloqueia o Viver Bem inteiro.

• Quando estarao prontos?

• O modelo tera conceito de "termo por empresa"?

• Haverá versionamento (versaoTermo)?

Blocker
3. Conceito de MedicaoRisco (estratificação clínica)

No Healthmap é risco de saúde populacional (BAIXO/MEDIO/ALTO). Na Tabia appointment-no-show-risk é risco operacional de no-show.

• Existe roadmap para endpoint de risco clínico?

• Posso usar PatientDTO.tabiaScore.severity como proxy?

• As faixas 0-0.33 / 0.33-0.66 / 0.66-1 são oficiais?

Blocker
4. Schema do endpoint POST /apiv1/surveydata/{uid}/answer

Mencionado no PDF mas ausente no OpenAPI. Crítico para registrar respostas.

• Enviar OpenAPI completo deste endpoint

• Suporta filtro por periodo?

Blocker
5. Documentação dos endpoints de WhatsApp

/apiv1/whatsapp-management e /apiv1/whatsapp/logs aparecem no PDF mas não no OpenAPI.

• Schema completo

• Como substitui o fluxo CRM Dynamics → BLIP atual?

• Templates/flows customizaveis?

Blocker
6. Modelo de Event customizado para Alertas de Internação

Healthmap tem endpoint dedicado. Tabia sugere usar Event com formData. Precisamos definir:

• Categoria fixa "INTERNACAO" pre-cadastrada?

• Campos custom obrigatorios: unidade, motivo, idGestor

• Qual integração alimenta os eventos (HIS hospitalar)?

Alta
7. Enum completo de tipos de questão (Survey.field.type)

Precisamos da lista oficial para criar tabela de-para com os tipos Healthmap (SIM_NAO, MULTIPLA_ESCOLHA, TEXTO_LIVRE, NUMERO, DATA).

• Enviar enum oficial de field.type

Alta
8. Enum completo de roles e rolesInOrganizations

Para mapear tipos Healthmap (PROFISSIONAL, GESTOR, NAVEGADOR, AUDITOR).

• Lista oficial de roles e rolesInOrganizations

• Como criar role custom se necessário?

Alta
👨‍💻 Decisões técnicas para alinhar com o Dev Lead 12 itens
1. Estrutura do adapter como Library ODC

Decisão já tomada: adapter será Tabia_ACL_Lib (Library ODC) consumindo REST do OpenAPI Tabia.

• Nome final da Library

• Namespace/stage (dev/test/prd)

• Quem possui? Time CBV ou time novo?

• Relacionamento com ModeloAssistencial_Lib atual — deprecar ou manter em paralelo durante transição?

Alta
1b. Consume REST API do OpenAPI Tabia no Service Studio

Importar openapi-careos-simple.json direto gera Structures + Client Actions automaticamente.

• Validar se o OpenAPI importa limpo (pode ter warnings por map<string,object> no Java)

• Endpoints ausentes no OpenAPI (/surveydata/{uid}/answer, WhatsApp) — criar manualmente

• Estratégia de atualização: re-import periodico ou manual?

Alta
2. Entity de mapeamento de IDs dentro da Library

Entity CBV_TabiaIdMapping com campos: EntityType (Patient|User|Survey|Task), CBV_Id, Tabia_Id, CreatedAt, LastSyncAt.

• Entity pública ou privada da Library? Privada — acesso só via Server Actions

• Indexes: (EntityType, CBV_Id) e (EntityType, Tabia_Id)

• Timer de sync inicial: popular via consulta combinada (CPF HM → email Tabia)

Alta
3. Feature flags via Site Property (Strangler Fig em ODC)

Em ODC, feature flags são Site Properties ou External Configurations.

• Criar UseTabiaFor_Beneficiario, UseTabiaFor_Questionario, etc (boolean)

• Server Action ShouldUseTabia(endpoint) centraliza a decisão

• Rollback = trocar flag (não precisa re-deploy)

Alta
4. Onde armazenar campos sem equivalente

Campos como idExterno, respostaTexto, respostaValor, idDispositivoIntegracao precisam de destino.

• Usar person.extensions (map<string,string>)?

• Tabela paralela no CBV?

• Padronizar chaves (cbv.idExterno, cbv.dispositivoId)

Media
5. Tratamento de Plano de Ação (1 entidade → 4 na Tabia)

Healthmap = PlanoAcao. Tabia = Pathway + ICP + Task + Event.

• Qual endpoint corresponde ao "plano" conceitual? individual-care-plan

• Como unir Task + Event em uma visao? Cache/composite service?

• Templates de Pathway vao ser replicados?

Media
6. Paginação — base 1 → base 0

Healthmap: página=1. Tabia: page=0. Atencao com off-by-one.

• ACL sempre converte? Expor nova API em base 0 para CBV?

• offset → page * size

Media
7. Normalizacao de datas

Healthmap usa ISO 8601 com horario. Tabia mistura ISO + epoch int64.

• Conversao no ACL sempre para ISO?

• Timezones (timeZoneId aparece em Event)

Media
8. Tabela de-para de enums de status

Criar biblioteca de conversao bidirecional:

• ATIVO/INATIVO ↔ deathDate/blocked/status=OPEN|CLOSED

• PENDENTE/EM_PROGRESSO/CONCLUIDO ↔ closed/accomplished/progress

• BAIXO/MEDIO/ALTO ↔ faixas de tabiaScore.severity

Media
9. API Key via ODC Secret (PRD/staging)

Em ODC: Settings > Secrets (ou External Configurations) — não vai para o código.

• Secrets por stage: Tabia_ApiKey_Dev, Tabia_ApiKey_Test, Tabia_ApiKey_Prd

• Acesso via GetEnvironmentSecret("TABIA_API_KEY")

• Rotacao — politica com Tabia

• Remover toda logica atual de OAuth2 token + refresh scheduler

Media
9b. Base URL por ambiente (ODC Configuration)

Dois hosts: https://staging.tabia.health (HMG) e https://app.tabia.health (PRD).

• Usar Effective URL dos REST API consumers no Service Studio

• Ou Site Property Tabia_BaseUrl

• Garantir que dev aponta para staging automaticamente

Media
10. Observabilidade no ODC Monitoring

Usar o ODC Portal > Monitoring já existente.

• LogMessage estruturado em cada mapper (payload HM in/out + tempo)

• Usar TraceId para correlacionar chamadas

• Alertas via Monitor > Custom

• Não usa stack externa (Splunk/Datadog) — já vem no ODC

Baixa
11. Testes unitarios dos Mappers (BDD Framework)

ODC tem BDD Framework nativo para testar Server Actions.

• Criar app de testes Tabia_ACL_Lib_Tests

• Cenarios: Patient valido → Beneficiário, Patient sem CPF, status mapping, etc

• Rodar no pipeline via LifeTime Plugins

Baixa
12. Anexos de Ação (realizarAcao.arquivos[])

Healthmap aceita array de arquivos binarios. Tabia não tem equivalente direto no TaskStatusDTO.

• Investigar NoteDTO.file (StorageFileDTO) na Tabia

• Talvez usar File Storage do próprio ODC e referenciar por URL

Baixa
13. Timeouts e retry dos Client Actions REST

Por padrao ODC usa timeout default. Chamadas Tabia podem precisar ajuste.

• Configurar timeout por endpoint (ex: /event?patients=... pode ser lento)

• Retry policy (exponential backoff) via BPT ou logica manual

• Circuit breaker para degradacao graciosa

Baixa
14. Permissoes e roles no ODC

A Library não expoe tela, mas os apps consumidores precisam permissao.

• Definir se a Library será pública na Forge interna

• Roles de quem pode chamar cada Server Action exposta
