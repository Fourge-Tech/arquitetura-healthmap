  • A API existe, está no ar e responde corretamente — retorna 401 sem auth, 403 com key inválida.

  • Swagger UI não é público — precisa autenticação para acessar.

🔐 Teste 2 — Comportamento da autenticação

CenárioHTTPInterpretação
GET /apiv1/patient?size=1
sem header api-key		 401 Unauthorized — servidor exige autenticação. Confirma endpoint existe.
GET /apiv1/patient?size=1
com api-key: invalid-test-key-123		 403 Forbidden — servidor recebeu a key mas não reconheceu. Diferenciação 401 vs 403 ajuda no tratamento de erro do adapter.
Para o Dev Lead: O tratamento de erro do Tabia_ACL_Lib deve diferenciar:
  • 401 = API Key ausente → alertar DevOps, secret pode ter sumido
  • 403 = API Key errada/expirada → trocar chave
  • 429 = rate limit (não testado) → backoff
  • 5xx = erro Tabia → retry + circuit breaker

🛠️ Como reproduzir os testes (para o Dev Lead)

Todos os comandos abaixo rodam em qualquer terminal com curl:

# 1. Conectividade básica
curl -s -o /dev/null -w "HTTP %{http_code} | %{time_total}s\n" https://staging.tabia.health/

# 2. Chamar API sem key (deve dar 401)
curl -s -i https://staging.tabia.health/apiv1/patient?size=1 | head -5

# 3. Chamar com key inválida (deve dar 403)
curl -s -i -H "api-key: test-fake-key" https://staging.tabia.health/apiv1/patient?size=1 | head -5

# 4. Teste de latência (5 amostras)
for i in 1 2 3 4 5; do
  curl -s -o /dev/null -w "  $i: %{time_total}s\n" https://staging.tabia.health/apiv1/patient?size=1
done

# 5. CORS preflight
curl -s -i -X OPTIONS \
  -H "Origin: https://meu-dominio.com" \
  -H "Access-Control-Request-Method: GET" \
  -H "Access-Control-Request-Headers: api-key" \
  https://staging.tabia.health/apiv1/patient

# 6. Depois que tiver API Key real:
curl -s -H "api-key: CHAVE_REAL_AQUI" https://staging.tabia.health/apiv1/patient?size=1 | jq
