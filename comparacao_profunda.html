Comparação Profunda — Healthmap x Tabia (CareOS)

Análise campo-a-campo de request/response — baseada na Postman Collection (uso real Healthmap) e OpenAPI CareOS (Tabia)
Legenda Resumo Auth Beneficiário Usuário Termos Risco Plano de Ação Questionários Internação Externos
🔤 Glossário de correlacoes PT → EN (nomes parecidos) — muitos "sem equivalente" são só traduções ▼
Contexto: Healthmap usa nomenclatura em portugues; Tabia (CareOS) usa ingles + convencoes healthtech internacionais (FHIR, HL7). Muitos campos que parecem "ausentes" na Tabia são apenas traduções ou sinonimos técnicos — ou estão em uma estrutura aninhada diferente.

Convencoes de nomenclatura

Healthmap — padroes

  • camelCase em portugues: idCliente, numeroCarteirinha, dataAgendamento
  • prefixo "id" para chaves estrangeiras: idCliente, idProfissional, idQuestionario
  • status como string UPPER_CASE: "ATIVO", "PENDENTE", "EM_PROGRESSO"
  • tipo como string UPPER_CASE: "BENEFICIARIO", "PROFISSIONAL"
  • DTOs sufixados com "DTO": BeneficiarioDTO, PlanoAcaoDTO
  • datas em ISO com hora: "2026-03-15T10:30:00"

Tabia — padroes

  • camelCase em ingles: patientId, medicalRecordId, targetDate
  • referência objeto: patient: {id, name} em vez de patientId
  • status como enum formal: PLANNED, STARTED, ACCOMPLISHED, NOT_ACCOMPLISHED
  • booleanos em vez de status: closed, accomplished, archived, blocked
  • DTOs sufixados com "DTO": PatientDTO, TaskDTO
  • datas misturam ISO + epoch int64: "2026-03-15" ou 1742040600000
  • sufixo "At" para timestamps: createdAt, finishedAt, deliveredAt, readAt
  • prefixo "date" para datas puras: dateCreated, dateClosed, dueDate, birthDate

Resumo da análise de nomes:
  • ~75% dos campos tem correspondência direta por tradução (nome→fullName, telefone→phoneNumber, dataNascimento→birthDate)
  • ~15% tem correspondência estrutural (status string → booleans, idCliente → patient.id)
  • ~7% precisam de extensions (CPF, CNPJ, idExterno, idDispositivoIntegracao)
  • ~3% são conceitos realmente ausentes (Termos de Aceite, MedicaoRisco clínico)
  • Principal armadilha: CPF não é "campo esquecido" — é convencao brasileira sem padrao internacional. Precisa ser decisão explicita com Tabia
Legenda de status: DIRETO campo com equivalente exato (só renomear) TRANSFORMAR campo existe mas precisa conversao/mapeamento NOVO TABIA campo que a Tabia oferece a mais SEM EQUIV. campo Healthmap que a Tabia não tem CUSTOMIZAR depende de customizacao/extensions N/A não aplicavel

Principais achados:
  • Volume de dados: Tabia retorna muito mais informação por entidade (tabiaScore, pathways, ICDs, teams) — response mais rico, mas também mais complexo de consumir
  • Identificadores: Healthmap usa CPF como chave primaria de busca; Tabia usa id (int64) + email/cellphone. CPF não aparece no PatientDTO — precisa ficar em extensions ou campo custom
  • Enumeracoes: Healthmap usa strings livres ("ATIVO", "ALTO"); Tabia usa enums formais (PLANNED|STARTED|ACCOMPLISHED). Migração exige tabela de-para
  • Datas: Healthmap usa ISO 8601 com horario ("2026-01-01T10:00:00"); Tabia mistura ISO + epoch timestamps (int64). Precisa normalizar
  • Paginação: Healthmap = pageSize/offset/pageNumber; Tabia = size/page/number — field mapping trivial mas numerico
  • Termos de Aceite: Domain inteiro ausente na Tabia (4 endpoints) — BLOCKER
  • MedicaoRisco: Conceito completamente diferente (no-show vs saúde populacional) — BLOCKER
  • Plano de Ação: 1 entidade Healthmap → 4 entidades Tabia (Pathway+ICP+Task+Event) — REDESIGN

Tabelas de-para necessárias

  • ID mapping: cbv_to_tabia_patient_id, cbv_to_tabia_user_id, cbv_to_tabia_survey_id, cbv_to_tabia_task_id
  • Status mapping: ATIVO/INATIVO → deathDate + since (pacientes); ATIVO/INATIVO → OPEN/CLOSED/CLOSING/CREATING/TRANSFERRING (planos)
  • Tipo de questão: SIM_NAO, MULTIPLA, TEXTO, NUMERO → YES_NO, MULTIPLE_CHOICE, TEXT, NUMBER (confirmar enum Tabia)
  • Tipo de usuário: PROFISSIONAL, GESTOR, NAVEGADOR → roles[]
  • Prioridade: ALTA, MEDIA, BAIXA → tags ou categories
  • statusAcao: PENDENTE, EM_PROGRESSO, CONCLUIDO, NAO_REALIZADO → combinacoes de closed + accomplished
  • statusRisco: BAIXO, MEDIO, ALTO → faixas de tabiaScore.severity

Estratégia recomendada de adapter

  1. Criar Anti-Corruption Layer que expoe interface identica ao Healthmap atual para o CBV
  2. Internamente, traduzir requests/responses usando as tabelas de-para acima
  3. Manter CBV ignorante da mudanca — Healthmap pode ser desligado após o adapter estabilizar
  4. Implementar em ondas: (1) Auth+Patient, (2) Survey, (3) PlanoAcao, (4) Risco+Internação+Termos, (5) Mensageria
